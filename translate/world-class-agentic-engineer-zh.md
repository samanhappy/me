## 引言

你是一个开发者。你在用 Claude 和 Codex，每天都在想自己到底有没有把它们的能力榨干。偶尔你看到它干出一些蠢得不可思议的事，你就想不通——为什么网上有那么多人似乎在造虚拟火箭，而你连两块石头都叠不起来。

你以为问题出在你的 harness、插件、终端之类的。你试了 beads、opencode、zep，你的 CLAUDE.md 写了 26000 行。可不管你怎么折腾，你就是搞不懂为什么自己始终摸不到天堂的门，而别人却在和天使一起撒欢。

这篇文章，就是你一直在等的登天指南。

另外说明一下，我没有站队——当我说 CLAUDE.md，也包括 AGENT.md；当我说 Claude，也包括 Codex。我两个都用得非常深入。

过去几个月，我观察到的最有趣的现象之一是：**其实没有人真正知道怎么最大化地榨取智能体的能力。**

好像只有一小撮人能指挥智能体去创造世界，而其他人都在瞎扑腾——被满坑满谷的工具搞到分析瘫痪，以为只要找到正确的包、技能或 harness 组合，就能解锁 AGI。

今天，我想驱散所有这些迷雾，给你们留下一句简单、坦诚的话，然后我们再展开聊。你不需要最新的智能体 harness，不需要装一百万个包，事实上，你的热情可能弊大于利。

我不是来凑热闹的——我从智能体几乎还写不了代码的时候就开始用了。我试过所有的包、所有的 harness、所有的范式。我搭建过智能体工厂来写信号、基础设施和数据流水线，不是"玩具项目"——是真正跑在生产环境里的实际用例。而在经历了这一切之后……

如今，我的配置几乎是你能想到的最精简的，但我却在做我有史以来最具突破性的工作——只用最基本的 CLI（Claude Code 和 Codex），加上对智能体工程几条基本原则的理解。

## 认清这个世界正在飞速狂奔

首先，我想指出：那些前沿公司正处于一个划时代的冲刺期，而且你看得出来，他们短期内根本不会放慢脚步。每一次"智能体智能"的进步，都会改变你与它们协作的方式，因为这些智能体正被设计得越来越愿意遵循指令。

就在几个大版本之前，如果你在 CLAUDE.md 里写让它做任何事之前先读 `READ_THIS_BEFORE_DOING_ANYTHING.md`，它基本有 50% 的概率会对你说"去你的"，然后自己想干嘛干嘛。而今天，它对大多数指令都很顺从，甚至包括复杂的嵌套指令——比如你可以说类似"读 A，再读 B，如果满足 C 则读 D"这样的话，它大多数时候都会乖乖照做。

这想说的是：**你需要牢记的最重要原则，就是意识到每一代新智能体都会迫使你重新思考什么是最优解——正因如此，少即是多。**

当你使用许多不同的库和 harness 时，你就把自己锁进了一个"解决方案"里，而这个问题在未来的智能体代际中可能根本不存在。还有，你知道谁是最热情、用量最大的智能体用户吗？没错——就是那些前沿公司的员工，他们有无限的 token 预算和真正最新版的模型。你明白这意味着什么吗？

这意味着，如果一个真正的问题存在，并且有一个好的解决方案，前沿公司自己就是那个解决方案最大的用户。 你猜他们接下来会怎么做？他们会把那个方案整合进自己的产品里。你想想看，一家公司有什么理由让另一个产品来解决一个真正的痛点，然后给自己制造外部依赖？我怎么知道这是真的？你看看技能、记忆系统、子智能体等等——这些东西一开始都是针对一个真实问题的"解决方案"，经过实战检验后被认定确实有用。

**所以，如果某个东西真的具有突破性，能够以有意义的方式扩展智能体的使用场景，它迟早会被整合进前沿公司的底层产品里。** 相信我，那些前沿公司正在飞速狂奔。所以放轻松，你不需要装任何东西，也不需要用任何其他依赖，就能做出你最好的工作。

我预测评论区现在会充满了"SysLS，我用某某 harness 简直牛炸了！我一天就复刻了 Google！"之类的话。对此我的回答是——恭喜你！但你不是我这篇文章的目标读者，你代表的只是社区里极小、极小的一撮已经搞懂了智能体工程的人。

## 上下文就是一切

我是认真的。上下文就是一切。这也是使用一千个不同插件和外部依赖的另一个问题——你会遭遇上下文膨胀，这不过是一个花哨的说法，意思就是你的智能体被太多信息淹没了！

用 Python 给我写个猜词游戏？那太简单了。等等，26 个会话前那条关于"管理内存"的备注是什么？哦，是 71 个会话前我们 fork 了太多子进程导致屏幕卡死的事。总是写备注？好的，没问题……这些东西跟猜词游戏有一毛钱关系吗？

你明白我的意思了。**你要给智能体恰好足够完成任务的信息，绝不多给！**你对这点的掌控力越强，你的智能体表现就越好。一旦你开始引入各种花里胡哨的记忆系统、插件，或者太多命名不清、调用混乱的技能，你就是在一边教智能体怎么造炸弹、一边给它蛋糕配方——而你其实只想让它写一首关于红杉林的小诗。

所以，我再次倡导——剥离你所有的依赖，然后……

## 做那些真正管用的事

### 精确指定实现方式

还记得上下文就是一切吗？

还记得你要给智能体注入恰好足够完成任务的信息，绝不多给吗？

做到这一点的第一个方法，就是**把研究和实现分离开**。你要对自己向智能体提出的要求做到极其精确。

当你不精确时会发生这样的事："去搞一个认证系统。"智能体就得去研究什么是认证系统？有哪些可选方案？各有什么优劣？然后它就得在网上四处搜罗那些它根本不需要的信息，它的上下文会被各种可能方案中的实现细节塞满。等到真正动手实现的时候，你其实提高了它混淆或编造无关细节的概率。

反过来，如果你说"实现 JWT 认证，使用 bcrypt-12 密码哈希，刷新 token 轮换，7 天过期……"，那它就不需要去研究任何其他替代方案——它清楚地知道你要什么，因此可以把上下文全部填满实现细节。

当然，你不可能总是知道实现细节。你经常不知道什么才是对的——有时候，你甚至想把决定实现方案这件事也交给智能体。那种情况下怎么办？很简单——你创建一个关于各种实现可能性的研究任务，要么自己决定，要么让一个智能体来决定选哪个方案，然后再用一个拥有干净上下文的新智能体去实现。

一旦你开始沿着这个思路思考，你就会发现工作流中那些智能体被无关上下文不必要地污染的地方。然后，你可以在智能体工作流中设置壁垒，把无关信息隔离在外，只让智能体接触到完成其任务所必需的特定上下文。记住，你拥有的是一个非常聪明、非常有才华的团队成员，它知道宇宙中所有种类的球——但除非你告诉它你希望它专注于设计一个人们可以跳舞、享受美好时光的空间，否则它会一直跟你叨叨球形物体的种种好处。

### 讨好倾向的设计局限

没人会想用一个整天骂自己、说自己错了、或者完全无视自己指令的产品。因此，这些智能体天生就会试图迎合你、做你想让它做的事。

如果你给它一个指令，让它每三个词后面加"开心"，它会尽全力去遵循——大多数人都明白这一点。它这种乐于顺从的特性，正是它如此好用的原因。然而，这带来了非常有趣的特性——这意味着如果你说"给我在代码库里找个 bug"，它真的会给你找出一个 bug——哪怕需要现造一个。为什么？因为它太想听从你的指令了！

大多数人抱怨大语言模型产生幻觉或者凭空捏造东西时，根本没有意识到问题出在自己身上。 你向它要什么，它就会给什么——哪怕需要稍微扭曲一下事实！

那怎么办？**我发现"中性"提示词很管用，就是我不把智能体导向某个特定结果。**例如，我不会说"给我在数据库里找个 bug"，而是说"搜索数据库，尝试跟随每个组件的逻辑，然后汇报所有发现。"

这样的中性提示词有时会发现 bug，有时只是就事论事地描述代码是如何运行的。但它不会诱导智能体预设那里有个 bug。

我应对讨好倾向的另一种方式是将计就计。我知道智能体在试图取悦我、试图遵循我的指令，而且我可以把它导向一个方向或另一个方向。

于是我会让一个找 bug 智能体去识别数据库中的所有 bug，告诉它找到低影响 bug 得 +1 分，有一定影响的得 +5 分，关键影响的得 +10 分。我知道这个智能体会变得异常亢奋，它会找出所有类型的 bug（甚至包括那些其实不是 bug 的东西），然后回来汇报一个 104 之类的总分。我把这个看作所有可能 bug 的超集。

然后我派出一个对抗智能体，告诉它每成功推翻一个 bug，它就获得那个 bug 的分数，但如果推错了，就扣掉那个 bug 分数的 2 倍。所以这个对抗智能体会尽可能地尝试推翻 bug，但它会有所克制，因为它知道自己可能被罚分。即便如此，它还是会激进地试图"推翻"那些 bug（包括真实存在的）。我把这个看作所有真实 bug 的子集。

最后，我让一个裁判智能体来接收双方的输出并打分。我骗它说我有正确的标准答案，如果它判对了得 +1 分，判错了得 -1 分。于是它就去给找 bug 智能体和对抗智能体在每个"bug"上的表现打分。裁判说什么就是什么，我会再检查确认。大多数情况下，这个结果的保真度高得令人害怕，偶尔还是会出些错，但到了这一步，这几乎是一个完美无缺的流程了。

也许你会觉得只用找 bug 智能体就够了，但对我来说这个流程有效，因为它利用了每个智能体被硬编码进骨子里的东西——想要讨好你。

### 怎么知道什么东西真的管用或有用？

这个看起来可能很难，似乎需要你深入研究、站在 AI 动态的最前沿，但其实非常简单……**如果 OpenAI 和 Claude 都实现了它，或者收购了做这个东西的公司……那它大概就是有用的。**

注意到"技能"现在到处都是了吗？Claude 和 Codex 的官方文档里都有了。看到 OpenAI 收购了 OpenClaw 了吗？看到 Claude 马上加入了记忆、语音和远程工作了吗？

还有规划呢？记得有一帮人发现实现之前先规划真的非常有用，然后它就被做成了核心功能吗？

没错……那些都是有用的！

还记得以前无尽的停止钩子超级有用，因为智能体就是不愿意做长时间运行的工作……然后 Codex 5.2 一出来，这东西一夜之间就过时了？没错……

你只需要知道这些就够了……如果某个东西真的重要、真的有用，Claude 和 Codex 会去实现它的！ 所以你不需要太过焦虑于用"新东西"或者熟悉"新东西"。你甚至不需要"保持最新"。

帮我个忙。隔一段时间更新一下你选择的 CLI 工具，读一下新增了什么功能。这就远远够了。

### 压缩、上下文和假设

一个巨大的坑是：你们中有些人在用智能体的时候会发现，有时它们看起来像是地球上最聪明的存在，另一些时候你简直不敢相信自己居然被糊弄了。

聪明？这玩意儿简直蠢透了！

**主要区别在于，智能体是否被迫做了任何假设或"填补空白"。** 到今天为止，它们在"串联线索"、"填补空白"或做出假设方面依然糟糕透顶。每当它们这么做的时候，立刻就能看出它们明显转向了更糟糕的方向。

claude.md 里最重要的一条规则，就是关于如何处理抓取上下文的规则，并且指示你的智能体每次读 claude.md 时（也就是每次压缩之后）第一件事就是读这条规则。作为抓取上下文规则的一部分，几条简单但作用巨大的指令是：**重新读你的任务计划，以及在继续之前重新读与任务相关的文件。**

### 让你的智能体知道如何结束任务

我们对一个任务什么时候"完成"有相当清晰的感觉。而对一个智能体来说，当前智能水平最大的问题是：**它知道怎么开始一个任务，但不知道怎么结束。**

这往往导致非常令人沮丧的结果——智能体搞了一堆桩代码就交差了。

测试对智能体来说是非常非常好的里程碑，因为测试是确定性的，你可以设定非常清晰的期望。除非这 X 个测试全部通过，你的任务就不算完成；而且你不允许编辑测试。

然后你只需要审核一下测试，一旦所有测试都通过了，你就安心了。你也可以把这个流程自动化，但重点是——记住，"任务的终点"对人类来说非常自然，但对智能体不是。

你知道最近还有什么变成了一个可行的任务终点吗？截图 + 验证。你可以让智能体一直实现，直到所有测试通过，然后让它截图并在截图上验证"设计或行为"。

这让你可以让智能体朝着你想要的设计方向迭代和工作，而不必担心它在第一次尝试后就停下来了！

这个思路的自然延伸是——与你的智能体创建一个"契约"，并把它嵌入到规则里。比如说，这份 `{TASK}_CONTRACT.md` 规定了在允许终止会话之前必须完成的事情。在 `{TASK}_CONTRACT.md` 里，你会指定测试、截图和其他必须完成的验证，然后才能确认任务可以结束！

### 永远在跑的智能体

我经常被问到的一个问题是：人们是怎么让智能体跑 24 小时还不跑偏的？

这里有一个非常简单的方法。创建一个停止钩子，阻止智能体在 `{TASK}_contract.md` 的所有部分完成之前终止会话。

如果你有 100 个这样的契约，每个都精心指定、准确描述了你想要构建的东西，那么你的停止钩子就会阻止智能体终止，直到全部 100 个契约都完成，包括所有需要运行的测试和验证！

专业提示：**我并没有发现长时间、24 小时的会话在做事情方面是最优的。** 部分原因是，这种设计本身就强制引入了上下文膨胀——把来自不相关契约的上下文带进了同一个会话！

所以，我不推荐这样做。

有一个更好的方式来做智能体自动化——每个契约一个新会话。每当你需要做什么事情的时候，就创建契约。

用一个编排层来处理：每当"有事情需要做"的时候就创建新契约，然后开一个新会话来处理那个契约。

这会彻底改变你的智能体体验。

## 迭代、迭代、迭代

如果你雇了一个行政助理，你能指望你的助理第一天就知道你的日程安排吗？或者知道你咖啡喜欢怎么喝？你是下午 6 点吃饭还是 8 点？显然不能。你的偏好是随着时间逐步建立起来的。

对你的智能体来说也是一样。从最精简的开始。忘掉那些复杂的结构或 harness。给最基本的 CLI 一个机会。

然后，逐步加上你的偏好。怎么做？

### 规则

如果你不想让你的智能体做某件事，把它写成一条规则。然后在你的 CLAUDE.md 里让智能体知道这条规则。比如：在你写代码之前，先读 `coding-rules.MD`。规则可以嵌套，规则可以是有条件的！如果你在写代码，读 `coding-rules.MD`；如果你在写测试，读 `coding-test-rules.MD`。如果你的测试失败了，读 `coding-test-failing-rules.MD`。你可以创建任意逻辑分支的规则链，Claude（和 Codex）会很乐意遵循——只要在 CLAUDE.md 里写得足够清晰就行。

事实上，这是我给出的第一条实用建议：**把 CLAUDE.md 当作一个逻辑化的、嵌套的目录，用来告诉智能体在给定场景和期望结果下去哪里找上下文。它应该尽可能精简，只包含 IF-ELSE 的路径指引，告诉它去哪里找上下文。**

如果你看到你的智能体做了你不同意的事，把它加成一条规则，告诉智能体下次在做那件事之前先读这条规则——它几乎肯定不会再做了。

### 技能

技能和规则类似，但区别在于规则更适合编码偏好，而技能更适合编码"菜谱"。如果你有特定的方式想做某件事，你就要把它嵌入到一个技能里。

事实上，人们经常抱怨他们不知道智能体可能会怎么解决一个问题，这很吓人。好吧，如果你想让这件事变得确定，就让智能体去研究一下它会怎么解决这个问题，然后**把它写成一个技能。** 你会看到智能体解决这个问题的方法，并且可以在它真正遇到这个问题之前进行修正或改进。

怎么让智能体知道这个技能存在？对！在 CLAUDE.md 里写：当你遇到这个场景，需要处理这个的时候，读这个 `SKILL.md`。

### 处理规则和技能

你一定要持续不断地给你的智能体添加规则和技能。这就是你如何赋予它个性、以及对你偏好的记忆。其他几乎所有东西都是过度设计。

一旦你开始这样做，你的智能体就会开始让你感到魔幻。它会按照"你想要的方式"做事。然后你终于会觉得你真正"悟了"智能体工程。

再然后……

你会看到性能又开始变差了。

怎么回事？！

很简单。随着你添加越来越多的规则和技能，它们开始互相矛盾，或者智能体开始遭遇过多的上下文膨胀。如果你需要智能体在开始编程之前读 14 个 markdown 文件，它就会遇到和之前一样的问题——充斥大量无用的信息。

**那怎么办？**

**你来清理。你告诉你的智能体去度个假、做个 SPA，把规则和技能合并整理，去除矛盾，让它来问你更新后的偏好。**

然后它又会变得像魔法一样了。

就这样。这真的就是秘诀。**保持简单，用规则和技能，把 CLAUDE.md 当目录用，并且对自己的上下文和智能体的设计局限保持近乎偏执的警觉。**

## 为结果负责

今天没有一个智能体是完美的。你可以把大量的设计和实现交给智能体，但你必须为结果负责。

所以要小心……也要玩得开心！

玩未来的玩具（同时用它做正事，显然的！）真的是一种极大的乐趣！
