驾驭工程：在智能体优先的世界中发挥 Codex 的能力
================================================

作者：Ryan Lopopolo，技术团队成员

在过去的五个月里，我们的团队一直在进行一项实验：用 **0 行手写代码**构建并交付一款软件产品的内部测试版。

这款产品拥有日常内部用户和外部 alpha 测试者。它会发布、部署、出故障、被修复。不同之处在于，每一行代码——应用逻辑、测试、CI 配置、文档、可观测性以及内部工具——全部由 Codex 编写。我们估算，构建这个产品所花的时间大约是手写代码所需时间的 1/10。

**人类掌舵，智能体执行。**

我们有意选择了这个约束条件，以便构建出能将工程效率提升数个数量级所必需的东西。我们只有几周时间来交付最终达到百万行代码的产品。要做到这一点，我们需要理解：当一个软件工程团队的主要工作不再是编写代码，而是设计环境、明确意图，并构建能让 Codex 智能体可靠工作的反馈闭环时，到底会发生什么变化。

这篇文章讲述的是我们用一个智能体团队从零开始构建一款全新产品所获得的经验——什么会出问题，什么会叠加放大，以及如何最大化我们唯一真正稀缺的资源：**人类的时间和注意力**。

从一个空的 git 仓库开始
-----------------------

第一次提交到一个空仓库发生在 2025 年 8 月下旬。

最初的脚手架——仓库结构、CI 配置、格式化规则、包管理器设置和应用框架——由 Codex CLI 使用 GPT-5 生成，并参考了一小组现有模板。甚至连指导智能体如何在仓库中工作的初始 `AGENTS.md` 文件本身也是由 Codex 编写的。

没有任何预先存在的人类编写代码来锚定这个系统。从一开始，这个仓库就是由智能体塑造的。

五个月后，仓库包含大约百万行代码，涵盖应用逻辑、基础设施、工具、文档和内部开发者实用程序。在此期间，大约有 1,500 个 Pull Request 被打开并合并，背后只有一个小型团队——三名工程师驱动 Codex。这相当于每位工程师每天平均 3.5 个 PR 的吞吐量，而且令人惊讶的是，随着团队增长到现在的七名工程师，吞吐量反而**提高了**。更重要的是，这并非为了产出而产出：该产品已经被数百名内部用户使用，其中包括每天的深度内部用户。

在整个开发过程中，人类从未直接贡献任何代码。这成为了团队的核心理念：**不写任何手写代码**。

重新定义工程师的角色
--------------------

手写代码的缺失**引入了一种不同的工程工作，聚焦于系统、支撑框架和杠杆效应**。

早期的进展比我们预期的要慢，不是因为 Codex 能力不够，而是因为环境定义不足。智能体缺乏实现高层目标所需的工具、抽象和内部结构。我们工程团队的主要工作变成了**让智能体能够做有用的工作**。

在实践中，这意味着以深度优先的方式工作：将更大的目标分解为更小的构建块（设计、代码、审查、测试等），提示智能体构建这些块，然后用它们来解锁更复杂的任务。当某件事失败时，解决方案几乎从来不是"再努力一点"。因为取得进展的唯一途径是让 Codex 来完成工作，人类工程师总是介入任务并问自己：**"缺少什么能力？我们如何让这个能力对智能体既可理解又可强制执行？"**

人类几乎完全通过提示词与系统交互：工程师描述一个任务，运行智能体，并允许它打开 Pull Request。为了推动 PR 完成，我们指示 Codex 在本地审查自己的更改，请求额外的特定智能体审查（包括本地和云端），回应任何人类或智能体给出的反馈，并在循环中迭代，直到所有智能体审查者都满意为止（这实际上就是一个 [Ralph Wiggum 循环](https://ghuntley.com/loop/)，即"重复尝试直到成功"的循环策略）。Codex 直接使用我们的标准开发工具（`gh`、本地脚本和嵌入仓库的技能）来收集上下文，而无需人类在 CLI 中复制粘贴。

人类**可以**审查 Pull Request，但并非必须。随着时间的推移，我们几乎将所有审查工作都推向了智能体对智能体的模式。

提高应用的可理解性
------------------

随着代码吞吐量的增加，我们的瓶颈变成了人类的 QA 容量。因为固定的约束始终是人类的时间和注意力，我们一直在努力通过让应用 UI、日志和应用指标本身对 Codex 直接可读，来为智能体增加更多能力。

例如，我们让应用可以在每个 git worktree 中启动，这样 Codex 就能为每个变更启动并驱动一个独立实例。我们还将 Chrome DevTools 协议接入到智能体运行时中，并创建了用于处理 DOM 快照、截图和导航的技能。这使 Codex 能够复现 bug、验证修复，并直接推理 UI 行为。

![标题为"Codex 通过 Chrome DevTools MCP 驱动应用来验证其工作"的图表。Codex 选择目标，在触发 UI 路径前后对状态进行快照，通过 Chrome DevTools 观察运行时事件，应用修复，重启，并循环重新运行验证直到应用正常。](https://images.ctfassets.net/kftzwdyauwt9/1Gu58eNlqDEuITmbqJDmq9/1e2e62f7e15fb16d2da0da5407240564/fig_1__codex_drives_the_app_.png?w=3840&q=90&fm=webp)

我们对可观测性工具做了同样的事。日志、指标和链路追踪通过一个本地可观测性技术栈暴露给 Codex，该技术栈对每个 worktree 是临时的。Codex 在应用的完全隔离版本上工作——包括其日志和指标，一旦任务完成就会被拆除。智能体可以使用 LogQL 查询日志，使用 PromQL 查询指标。有了这些上下文，像"确保服务启动在 800 毫秒内完成"或"这四个关键用户旅程中没有任何 span 超过两秒"这样的提示词就变得可操作了。

![标题为"在本地开发环境中为 Codex 提供完整的可观测性技术栈"的图表。应用将日志、指标和链路追踪发送到 Vector，Vector 将数据扇出到一个可观测性技术栈，包含 Victoria Logs、Metrics 和 Traces，各自通过 LogQL、PromQL 或 TraceQL API 查询。Codex 使用这些信号进行查询、关联和推理，然后在代码库中实施修复，重启应用，重新运行工作负载，测试 UI 旅程，并在反馈循环中重复。](https://images.ctfassets.net/kftzwdyauwt9/4Xr18TZ5G4Bh8zIgsTFIVK/f7ae689ddd8c31664e39d809b0973425/OAI_Harness_engineering_Giving_Codex_a_full_observability_stack_desktop-light__1_.svg?w=3840&q=90)

我们经常看到单次 Codex 运行在单个任务上持续工作长达六个小时以上（通常是在人类睡觉的时候）。

我们让仓库知识成为权威记录系统
---------------------------

上下文管理是让智能体在大型复杂任务上高效工作的最大挑战之一。我们最早学到的经验之一很简单：**给 Codex 一张地图，而不是一本 1,000 页的说明书。**

我们尝试过"一个大 [`AGENTS.md`](https://agents.md/)"的 approach。它不出所料地失败了：

* **上下文是稀缺资源。** 一个巨大的指令文件会挤占任务、代码和相关文档的空间——于是智能体要么遗漏关键约束，要么开始为错误的约束做优化。
* **过多的指导变成** ***无效指导***。当每件事都"重要"时，就没有什么是重要的了。智能体最终会进行局部模式匹配，而不是有目的地导航。
* **它迅速腐化。** 一本庞大的手册会变成陈旧规则的墓地。智能体无法分辨哪些仍然有效，人类停止维护它，文件悄然变成了一个诱人的陷阱。
* **难以验证。** 一个单一的文本块不利于机械化检查（覆盖率、时效性、所有者、交叉链接），因此信息偏移是不可避免的。

所以，我们不再把 `AGENTS.md` 当作百科全书，而是把它当作**目录**。

仓库的知识库存在一个结构化的 `docs/` 目录中，将其视为权威记录系统。一个简短的 `AGENTS.md`（大约 100 行）被注入到上下文中，主要作为地图，指向更深层的真相来源。

仓库内知识存储布局。

设计文档被编目和索引，包括验证状态和一套定义智能体优先操作原则的核心信念。[架构文档](https://matklad.github.io/2021/02/06/ARCHITECTURE.md.html)提供了领域和包分层的顶层地图。质量文档对每个产品领域和架构层进行评分，跟踪随时间变化的差距。

计划被视为一等产物。轻量级的临时计划用于小型变更，而复杂的工作则被捕获在[执行计划](https://cookbook.openai.com/articles/codex_exec_plans)中，包含进度和决策日志，并签入仓库。活跃计划、已完成计划和已知技术债务全部进行版本管理并放在一起，使智能体能够在不依赖外部上下文的情况下运作。

这实现了渐进式披露：智能体从一个小的、稳定的入口点开始，并被教导下一步该去哪里查找，而不是一开始就被信息淹没。

我们通过机械化手段强制执行这一点。专用的 linter 和 CI 作业验证知识库是否是最新的、交叉链接的、结构正确的。一个定期运行的"文档园艺"智能体扫描陈旧或过时的文档（未能反映真实代码行为的），并打开修复性 Pull Request。

智能体可理解性是目标
--------------------

随着代码库的演进，Codex 用于设计决策的框架也需要随之演进。

因为仓库完全由智能体生成，它首先为 **Codex 的可理解性**而优化。就像团队致力于提高代码对新入职工程师的可导航性一样，我们人类工程师的目标是让智能体能够**直接从仓库本身**推理整个业务领域。

从智能体的视角来看，任何在运行时无法在上下文中访问到的东西，实际上就是不存在的。存在于 Google Docs、聊天记录或人们头脑中的知识对系统来说是不可访问的。仓库本地的、版本化的产物（例如代码、markdown、模式定义、可执行计划）是它能看到的一切。

![标题为"智能体知识的边界：Codex 看不到的东西就不存在"的图表。Codex 的知识被表示为一个有界的圆圈。下方是不可见知识的示例——Google Docs、Slack 消息和隐性人类知识。箭头指示要将这些信息对 Codex 可见，就必须将其编码为 markdown 存入代码库。](https://images.ctfassets.net/kftzwdyauwt9/7uWHsJIC6o3uQPsnQ2Avz9/8be3e321892054bd215afb2b250a176a/OAI_Harness_engineering_The_limits_of_agent_knowledge_desktop-light.png?w=3840&q=90&fm=webp)

我们认识到，随着时间的推移，需要将越来越多的上下文推入仓库。那次让团队在架构模式上达成一致的 Slack 讨论？如果智能体无法发现它，它就是不可理解的——就像三个月后入职的新员工无从知晓一样。

给 Codex 更多上下文意味着组织和暴露正确的信息，让智能体能够对其进行推理，而不是用临时的指令淹没它。就像你会向新队友介绍产品原则、工程规范和团队文化（包括 emoji 偏好）一样，把信息给到智能体会带来更好、更对齐的输出。

这个框架澄清了许多权衡。我们倾向于那些可以完全内化并在仓库内进行推理的依赖和抽象。通常被描述为"无聊"的技术，由于可组合性、API 稳定性以及在训练集中的代表性，往往更容易被智能体建模。在某些情况下，让智能体重新实现部分功能比绕过公共库中不透明的上游行为更划算。例如，我们没有引入一个通用的 `p-limit` 风格的包，而是实现了自己的 map-with-concurrency 辅助函数：它与我们的 OpenTelemetry 插桩紧密集成，拥有 100% 的测试覆盖率，并且行为方式完全符合我们运行时的预期。

将系统中更多部分拉入智能体可以检查、验证和直接修改的形式，会放大杠杆效应——不仅对 Codex 如此，对也在该代码库上工作的其他智能体（例如 [Aardvark](/index/introducing-aardvark/)）也是如此。

强制执行架构与品味
------------------

光靠文档无法保持一个完全由智能体生成的代码库的一致性。**通过强制执行不变量，而非微观管理实现方式，我们让智能体在不破坏基础的前提下快速交付。** 例如，我们要求 Codex [在边界处解析数据形状](https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/)，但不对如何实现做指令性规定（模型似乎喜欢 Zod，但我们并没有指定那个具体的库）。

智能体在边界严格、结构可预测的环境中最高效，因此我们围绕一个严格的架构模型构建了应用。每个业务领域被划分为一组固定的层，具有严格验证的依赖方向和一组合规的允许边。这些约束通过自定义 linter（当然也是 Codex 生成的！）和结构测试被机械化地强制执行。

下面的图表展示了规则：在每个业务领域内（例如应用设置），代码只能沿着一组固定的层"向前"依赖（类型 → 配置 → 仓库 → 服务 → 运行时 → UI）。横切关注点（认证、连接器、遥测、功能开关）通过一个单一的显式接口进入：Providers。其他任何方式都不被允许，并被机械化地强制执行。

![标题为"具有显式横切边界的分层领域架构"的图表。业务逻辑领域内部包含模块：类型 → 配置 → 仓库，以及 Providers → 服务 → 运行时 → UI，底部是应用装配 + UI。一个 Utils 模块位于边界之外，并输入到 Providers。](https://images.ctfassets.net/kftzwdyauwt9/4Rlip1H3T9apPlSmWs7Wr8/7708c176bfbe11951e06ad8e2b83bf01/OAI_Harness_engineering_Layered_domain_architecture_with_explicit_cross-cutting_boundries_desktop-light.png?w=3840&q=90&fm=webp)

这种架构通常是你有几百名工程师时才会考虑推迟实施的。而有了编程智能体，它就变成了早期的先决条件：**正是这些约束在不导致架构腐化或偏移的情况下，使得速度成为可能。**

在实践中，我们通过自定义 linter 和结构测试来强制执行这些规则，外加一小组"品味不变量"。例如，我们静态地强制执行结构化日志、模式和类型的命名规范、文件大小限制，以及带有自定义 lint 的特定平台可靠性要求。因为 lint 是自定义的，我们将错误消息写成可以向智能体上下文注入修复指令的形式。

在人类优先的工作流中，这些规则可能会显得迂腐或束缚手脚。而对于智能体，它们变成了倍增器：一旦编码，它们就可以在每一处同时生效。

与此同时，我们明确说明了约束在哪些地方重要，在哪些地方不重要。这类似于领导一个大型工程平台组织：在中心强制执行边界，在局部允许自治。你非常关心边界、正确性和可复现性。在这些边界之内，你允许团队——或智能体——在表达解决方案时拥有显著的自由度。

产生的代码并不总是符合人类的风格偏好，这没关系。只要输出是正确、可维护且对未来智能体运行可读的，它就达到了标准。

人类的品味被持续地反馈到系统中。 审查意见、重构 Pull Request 和面向用户的 bug 被捕获为文档更新，或直接编码到工具中。当文档不足以解决问题时，我们将规则提升为代码。

吞吐量改变合并哲学
------------------

随着 Codex 吞吐量的提高，许多传统的工程规范变得适得其反。

仓库以最小化的阻塞合并门禁运作。Pull Request 的存活时间很短。测试的不稳定性通常用后续运行来解决，而不是无限期地阻塞进展。在一个智能体吞吐量远超人类注意力的系统中，**修正代价低，等待代价高。**

这在低吞吐量环境中是不负责任的。但在这里，它往往是正确的权衡。

"智能体生成的"真正含义
---------------------

当我们说代码库由 Codex 智能体生成时，我们指的是代码库中的**一切**。

智能体产生：

* 产品代码和测试
* CI 配置和发布工具
* 内部开发者工具
* 文档和设计历史
* 评估框架
* 审查意见和回复
* 管理仓库本身的脚本
* 生产仪表板定义文件

人类始终处于闭环中，但工作在一个与我们过去不同的抽象层上。我们确定优先级，将用户反馈转化为验收标准，并验证结果。当智能体遇到困难时，我们将其视为一个信号：识别缺失了什么——工具、护栏、文档——并将其反馈到仓库中，而且始终是让 Codex 自己编写修复。

智能体直接使用我们的标准开发工具。它们拉取审查反馈，内联回复，推送更新，经常自己 squash 和合并 Pull Request。

不断提高的自主层级
----------------

随着越来越多的开发循环被直接编码到系统中——测试、验证、审查、反馈处理和恢复——仓库最近跨过了一个有意义的门槛：**Codex 可以端到端地驱动一个新功能。**

给定一个提示词，智能体现在可以：

* 验证代码库的当前状态
* 复现报告的 bug
* 录制演示故障的视频
* 实施修复
* 通过驱动应用验证修复
* 录制第二个视频演示解决方案
* 打开 Pull Request
* 回应智能体和人类的反馈
* 检测并修复构建失败
* 仅在需要判断时才升级给人类
* 合并变更

这种行为高度依赖这个特定仓库的结构和工具，不应假设在没有类似投入的情况下就能泛化——至少目前还不能。

熵与垃圾回收
-----------

完全的智能体自治也带来了全新的问题。 Codex 会复制仓库中已存在的模式——即使是不均衡或次优的模式。随着时间的推移，这不可避免地导致偏移。

最初，人类手动处理这些问题。我们的团队过去每个周五（一周的 20% 时间）都在清理"AI 垃圾代码"。不出所料，这无法规模化。

于是，我们开始将所谓的"黄金原则"直接编码到仓库中，并建立了一个定期运行的清理流程。这些原则是有明确观点的、可机械化执行的规则，旨在保持代码库对未来智能体运行的可读性和一致性。例如：（1）我们偏好共享工具包而非手写的辅助函数，以保持不变量集中化；（2）我们不"随意地"探测数据——我们要么在边界处验证，要么依赖带类型的 SDK，这样智能体就不会意外地基于猜测的数据结构来构建。按照固定频率，我们有一组后台 Codex 任务扫描偏离情况，更新质量评级，并打开有针对性的重构 Pull Request。其中大多数可以在不到一分钟内审查完成并自动合并。

这就像垃圾回收一样运作。技术债务就像一笔高息贷款：几乎总是更好的做法是持续地以小额增量偿还，而不是让它利滚利，然后在痛苦的爆发中处理。**人类品味被捕获一次，然后持续地在每一行代码上强制执行。** 这也让我们能够每天捕获和解决坏模式，而不是让它们在代码库中扩散数天或数周。

我们仍在学习的事情
----------------

这个策略到目前为止在 OpenAI 的内部发布和采用过程中运行良好。为真实用户构建真实产品有助于将我们的投入锚定在现实中，并引导我们走向长期可维护性。

我们还不了解的是，在一个完全由智能体生成的系统中，架构一致性在数年时间跨度内如何演化。我们仍在学习人类判断在哪些地方能带来最大的杠杆效应，以及如何编码这种判断使其产生复利效应。我们也不清楚随着模型能力持续提升，这个系统将如何演变。

已经变得清晰的是：**构建软件仍然需要纪律，但这种纪律更多地体现在支撑框架而非代码本身。** 保持代码库一致性的工具、抽象和反馈闭环变得越来越重要。

我们现在最困难的挑战集中在设计环境、反馈闭环和控制系统，以帮助智能体实现我们的目标：大规模地构建和维护复杂、可靠的软件。

随着像 Codex 这样的智能体在软件生命周期中承担越来越大的份额，这些问题将变得更加重要。我们希望分享一些早期经验，能帮助你思考将精力投向何处，以便[你可以专注于构建事物本身](/codex/)。
