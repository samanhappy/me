## 结果先说

2025 年 3 月 31 日创建仓库，到今天刚好一年。MCPHub 拿到了 2000+ star、248 fork、93 个 release。全程业余维护，白天上班，晚上和周末写代码。

这篇文章不是成功学，2000 star 在开源世界里算不上什么大数字。但作为一个全职打工人的 side project，这个过程中有一些决策和节奏上的经验，值得记下来。

## 为什么做这个

2024 年 11 月 25 日，Anthropic 发布了 MCP（Model Context Protocol）协议。这是一个让 AI 模型连接外部工具和数据的开放标准——简单说，就是给 AI 装"手"和"眼"。

我当时看完协议文档后的第一反应是：这东西会爆发，但有一个明显的缺口——当你接的 MCP server 越来越多，谁来管它们？

一个 MCP server 好管，十个呢？二十个呢？每个 server 有不同的启动方式、不同的配置格式、不同的认证机制。作为架构师，我很自然地想到这需要一个统一的管理层——一个 hub。

从有想法到动手写代码，中间隔了大概四个月。这四个月里 MCP 生态在快速成长，越来越多的 MCP server 被开发出来，"多 server 管理"的需求也越来越真实。2025 年 3 月底，我觉得时机到了。

## 第一周：疯狂迭代

4 月 13 日发布 v0.1.0，第一个正式版本。功能很简单：一个管理后台，能登录，能配置 MCP server，配置能持久化。

然后事情变得密集了。

4 月 14 日，v0.1.1 修复 bug。4 月 15 日，v0.2.0。4 月 17 日，v0.3.0。4 月 18 日——一天发了 6 个版本，从 v0.3.2 到 v0.3.8。

为什么这么快？因为早期用户开始来了。issue 一个接一个，有的是 bug，有的是功能需求，每一个都是真实场景。我当时的策略很简单：**用户报了就修，想到了就做，做完就发**。没有发布计划，没有里程碑，纯粹靠手感和反馈驱动。

回头看，这种节奏不可持续，但在项目初期，快速响应用户是建立信任的最有效方式。早期用户愿意容忍 bug，但不能容忍"没人理我"。

## 几个关键决策

### Market：一键安装 MCP Server（v0.4.0）

4 月 20 日，距离第一个版本发布仅一周，我做了一个现在回头看非常重要的决策：引入 Market。

Market 的逻辑很简单——用户不需要自己去找 MCP server 的配置方式，在 MCPHub 里浏览、一键安装就行。这把 MCPHub 从"给高级用户用的管理工具"变成了"任何人都能用的入口"。

这是我在做架构设计时常说的一句话的实践：**降低一步门槛，扩大十倍用户群**。

### Smart Routing：从管理工具到智能网关（v0.7.0）

5 月 28 日的 v0.7.0 引入了 Smart Routing——用语义搜索和 rerank 来自动发现最匹配的工具。

在此之前，MCPHub 本质上是个"配置管理中心"，帮你把多个 MCP server 管好。Smart Routing 之后，它变成了"智能网关"——AI agent 发出一个请求，MCPHub 能自动把请求路由到最合适的工具。

这个功能的灵感来自一个实际问题：当你接了二十几个 MCP server，里面可能有上百个 tool。把所有 tool 都暴露给 AI 模型？context window 装不下。只暴露一部分？那怎么选？

答案是：**让 AI 自己选**。用语义相似度找到最相关的几个 tool，只把它们交给模型。这就是 Smart Routing 做的事。

### 拥抱 AI 辅助开发（v0.10.0+）

到 v0.10.0（2025 年 10 月），项目的 PR 里开始出现大量 GitHub Copilot 的贡献。不是我一个人在用 Copilot，而是 Copilot 直接在帮我写 PR。

这是一种有趣的递归：一个服务于 AI 工具生态的项目，本身也在被 AI 工具加速开发。

我的经验是：对于功能边界清晰的任务（添加一个新的 API endpoint、写一组测试、重构一段配置逻辑），AI 辅助开发的效率提升是实打实的。但架构决策、功能取舍、优先级排序——这些仍然需要人来做。

### PostgreSQL 存储（v0.11.0）

2025 年 11 月的 v0.11.0 引入了 PostgreSQL 存储支持。在此之前，MCPHub 用的是文件存储。

做这个决策的原因很直接：开始有人想在生产环境跑 MCPHub 了。文件存储在单机场景下没问题，但一旦涉及多实例部署、数据备份、权限控制，数据库就是必须的。

这也标志着 MCPHub 从"个人工具"开始走向"可以部署在公司里的基础设施"。

## 功能演进的逻辑

回头看整个版本线，有一条比较清晰的路径：

1. **能跑起来**（v0.1-v0.3）：基础配置管理，证明概念可行
2. **连接生态**（v0.4-v0.5）：Market + Streamable HTTP，降低使用门槛
3. **变得聪明**（v0.6-v0.7）：多 session + Smart Routing，从管理工具升级为智能网关
4. **面向生产**（v0.8-v0.11）：OpenAPI 转换、OAuth、PostgreSQL，企业级能力
5. **精细运营**（v0.12）：活动追踪、向量搜索，为下一步提供数据支撑

每个阶段大概持续 2-3 个月。不是提前规划好的——更多是根据用户反馈和生态变化自然演化出来的。

## 社区运营

MCPHub 有一个 300 多人的微信群。说实话，社区运营不是我擅长的事。我的做法比较朴素：

- **Issue 不过夜**：GitHub issue 尽量当天回复，哪怕只是说一句"收到了，我看看"
- **用户需求分级**：bug 优先修，功能需求看是否通用
- **不承诺时间线**：业余维护最怕的就是 over-commit，告诉用户"我会做但说不好什么时候"
- **微信群主要用来收集需求**：深度讨论还是放 GitHub

300 人的群不大，但质量比数量重要。很多核心功能的灵感都来自群里的讨论。

## 节奏管理

全职上班 + 业余维护开源项目，时间管理是绕不过的话题。

我的策略是**接受不均匀**。有些周末能投入整整两天，有些周只能抽一两个小时处理 issue。不强求每周都有产出，但保持"活着"的信号——至少回复 issue，至少让用户知道这个项目没有被放弃。

平均下来大概是每周 1-2 个 release，但实际分布非常不均：前两周密集轰炸，之后趋于稳定，每隔几周会有一个大版本集中投入。

另一个经验是：**不要在工作日晚上做需要深度思考的工作**。白天上班已经消耗了决策资源，晚上适合做确定性高的任务——修 bug、写测试、改文档。架构设计和新功能留给周末精力充沛的时候。

## 运气和时机

诚实地说，MCPHub 的增长有很大的时机因素。

MCP 协议从发布到被广泛采用，经历了一个快速爬升期。2025 年 12 月 Anthropic 把 MCP 捐给 Linux Foundation 下的 AAIF，进一步确认了它的行业地位。在这个过程中，"MCP 生态工具"这个品类本身在膨胀，MCPHub 作为早期进入者吃到了红利。

如果我晚半年开始做，竞争环境会完全不同。

另外，解决一个"真实且通用"的问题也很关键。"多 server 管理"不是我创造出来的需求，是任何认真使用 MCP 的人都会遇到的问题。真实需求不需要你去教育用户——他们自己会找到你。

## 还没做好的事

说几个不足：

- **文档还是太差**：技术文档是我一直欠着的债，用户经常在群里问的问题其实文档应该覆盖
- **没有商业化探索**：2000 star 之后怎么走，我还没想清楚
- **测试覆盖率不够**：业余维护的副作用是"能跑就不动"，技术债在积累
- **国际化社区做得不够**：微信群覆盖中文用户，但海外社区的运营几乎为零

## 写在最后

一个人能做什么？

在 AI 辅助开发的时代，答案比以前要乐观得多。一个全职打工人，利用业余时间，一年内可以维护一个有 2000+ star 的开源项目，发布 93 个版本，服务几百个活跃用户。这在五年前是不太可能的。

但工具只是放大器。放大的是你对问题的理解、对时机的判断、对用户的尊重。这些东西没有捷径。

MCPHub 还在路上。下一步往哪走，我还在想。但回头看这一年，我觉得最对的一件事是：**在看到机会的时候动手了，而不是等到"准备好了"再开始**。

没有人准备好了。先做起来再说。
