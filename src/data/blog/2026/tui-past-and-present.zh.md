某天下午，我用 Claude Code 写一个用户模块。

它在终端里读取代码、编辑文件、运行测试。我在旁边看着文本流滚动——读了一个文件，改了几行，又去读另一个，回来再改，跑了一遍测试。全绿。

整个过程就在这个黑色终端窗口里。没有进度条，没有弹窗，没有"正在思考"的动画。文本一行行刷出来，活就干完了。

我盯着屏幕看了一会儿。不是因为结果——写个用户模块而已。而是这个交互方式本身让我觉得新鲜：一个 AI agent 在纯文本界面里自主完成了一连串操作，整个过程没有任何图形元素参与。

一个问题冒了出来：为什么 AI 工具偏偏选了 TUI 这种形态？

## 什么是 TUI

你打开电脑，双击图标，看到窗口、按钮、菜单——这是 GUI（Graphical User Interface，图形用户界面）。

你打开终端，输入命令，看到文本输出——这是 CLI（Command Line Interface，命令行界面）。

TUI（Text-based User Interface，文本用户界面）介于两者之间。它运行在终端里，但不只是"输一行命令、得一行结果"。它能画菜单、做表格、切窗口、响应键盘快捷键，甚至播放动画。只不过一切都是用文字字符"画"出来的。

你大概率用过的 TUI：vim、htop、top。你可能用过但不知道它叫 TUI 的：Lazygit、Ranger、Midnight Commander。

TUI 不是新东西。它比 Windows 还老。

## 前世：三次浪潮

### VT100：一切的起点

1978 年，DEC 公司推出 VT100 终端。

VT100 定义了"终端应该长什么样"。它引入的 ANSI 转义码——控制光标移动、文字颜色、屏幕清空的指令——今天仍然是每个终端模拟器遵循的标准。

你现在打开的每一个终端窗口，骨子里都是一台 VT100。

那时候没有"全屏界面"这个概念。终端就是终端，一行一行打字。但 ANSI 转义码给了开发者一个能力：把光标移到屏幕任意位置，在那里写一个字符，或者擦掉一块区域。

这个能力不起眼。但它为全屏文本界面铺了路。

### DOS 黄金时代

1987 年，Borland 推出 Turbo Pascal 和 Turbo C 的 IDE。

没见过这些工具的话，我帮你描述：蓝色背景上，白色窗口弹出下拉菜单，快捷键在不同面板间跳转，底部状态栏，编译速度快得离谱。

Borland 后来推出了 Turbo Vision 框架，把这些交互模式封装成可复用组件——窗口、菜单、对话框、按钮。开发者第一次能用搭积木的方式构建 TUI 应用。

同一时期，Norton Commander 定义了另一种经典模式：双栏文件管理器。左右两栏显示目录内容，F5 复制，F6 移动，F8 删除。这套操作逻辑影响了后来的无数工具，包括 Linux 上的 Midnight Commander，到今天仍有人在用。

那个年代的程序员不会觉得自己在用"落后"的东西。TUI 就是当时最先进的界面。

### Linux 与 ncurses

进入 Linux 时代，curses 库（和它的开源重写 ncurses）继承了 Turbo Vision 的衣钵。

ncurses 让 TUI 程序可以跨终端运行——不管你用的是 VT100、xterm 还是 GNOME Terminal，同一份代码都能正确显示。

Linux 上最广为人知的 TUI 大概是 `make menuconfig`，配置内核选项用的那个蓝色界面。mutt 是那个时代的经典邮件客户端。vim 和 emacs 则是编辑器之争的永恒主角。

这些工具有一个共同特点：学起来费劲，上手之后效率碾压大多数 GUI 同类。vim 的模式编辑、宏录制、文本对象操作，GUI 编辑器到今天也追不上。

## GUI 来了

1995 年，Windows 95 发布。2001 年，Mac OS X 面世。个人电脑的界面革命完成了。

用户想要窗口、按钮、鼠标点击。"好用"变成了"图形化"的同义词。TUI 退到了服务器管理员的 SSH 窗口和极客的终端里。

但 TUI 没死。vim、tmux、screen 到今天还有活跃的用户群体。一旦你用熟了，它们的效率确实比大多数 GUI 工具更高。

TUI 退居二线，没有退场。

## 为什么又火了

2020 年前后，情况变了。两股力量同时发力。

### 框架让 TUI 变简单了

Rust 生态的 Ratatui 是目前最活跃的 TUI 框架之一，GitHub 上超过 16000 颗 star。它用即时渲染模式：你逐帧告诉它"在屏幕的这个位置画什么字符"，它就画什么。没有复杂的继承体系，没有强制的架构模式。

Go 生态的 Bubble Tea 选了另一条路。它基于 Elm 架构，把应用拆成 Model（状态）、Update（事件处理）、View（渲染）三块。键盘按下是事件，窗口变化是事件，你响应事件来更新状态、重绘界面。这种函数式写法在 Go 社区很受欢迎。

Bubble Tea 背后的团队 charmbracelet 还围绕它搭了一整套终端美化工具链——语法高亮、Markdown 渲染、精致的表格和进度条。他们证明了终端界面也可以好看。

JS 生态有 Ink，Sindre Sorhus 做的。它让你用 React 的方式写 TUI——组件、状态、渲染，和写 React 网页的心智模型一样。前端开发者几乎零学习成本。

Rust 追求极致性能，Go 强调简洁优雅，JS 优先照顾开发者体验。三个框架指向同一个结论：**写一个好看的 TUI 应用，早已不再是 2005 年那种只有 C 语言老手才能搞定的苦差事。**

### AI 时代需要 TUI

框架进化让 TUI 变得"能写"。真正让它变得"该写"的，是 AI。

2024 年之后 AI 编程工具爆发。看看它们的形态：Claude Code 是 TUI，Aider 跑在终端里，Cursor 是 VS Code 的改造，Copilot Chat 嵌在编辑器里。

从零设计的独立 AI 编程工具，大部分选了 TUI。原因很直接。

值得一提的是，Claude Code 本身就是用 Ink 构建的——文章前面提到的那个 JS 框架。一个 AI 编程工具选了 TUI 形态，又选了 React 式的 TUI 框架，这个选择链条本身就说明了问题。

AI 的输出是文本。代码、解释、分析、建议，全是文本。终端天生为文本设计，没有多余的渲染开销，文本流直接刷在屏幕上。

AI 需要调用工具。运行 shell 命令、执行 git 操作、编译代码、查看日志——终端本身就是做这些事最自然的环境。一个跑在终端里的 AI agent 不需要绕道 GUI 去间接调用命令行，它直接就在命令行里。

开发者本来就住在终端里。一个能嵌入现有工作流的工具，比一个需要切换窗口、点按钮的工具摩擦小得多。

我日常用 Claude Code 的工作流是这样的：终端里输入一段需求，Claude 开始读取代码库、编辑文件、运行测试。整个过程我都在这一个终端窗口里看到反馈——它跑了什么命令，改了哪些文件，测试过没过。

如果这些发生在 GUI 里呢？各种面板和动画，不同标签页切来切去，等 loading 转圈。终端把这些全砍了。

**AI 需要的是文本输入输出和工具调用能力，不需要窗口、按钮和鼠标。终端恰好是满足这个需求的最小界面。**

## 写在最后

TUI 不会取代 GUI。浏览图片、复杂布局、给非技术用户用的软件，GUI 依然更合适。

回头看这篇文章的两条线索：框架作者们让 TUI 变得好写，AI 时代让 TUI 成了刚需。这两条线交汇在 2024 到 2026 年，于是有了这波浪潮。

我用 Claude Code 时最让我意外的，不是它能做什么，而是我几乎忘了自己在用一个"界面"。没有菜单栏，没有状态栏，没有设置页面。一行 prompt，一段输出。但它比我之前用过的任何 IDE 插件都高效。

**最好的界面，也许就是让你忘了界面存在的界面。** 而 TUI 恰好在这个时代找到了它的位置。

你现在的开发工具里，有几个是 TUI？
