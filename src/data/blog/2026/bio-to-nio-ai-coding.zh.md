## 写了十几年 Java，我被一个 IO 模型困住了

写了十几年 Java，我从没想过 BIO/NIO 这些 IO 模型的概念，会和我每天用 AI 写代码的方式产生关联。

毕竟一个是网络编程的底层原理，一个是 2026 年最火的开发方式。八竿子打不着。

但当我发现自己用 AI Coding 的效率越来越低时，我回头一看——我一直在用 **BIO 模式**。

不是比喻。是真的 BIO。

## 从 BIO 说起：一个线程阻塞一个连接

如果你写过 Java 网络编程，你一定知道 BIO（Blocking IO）的模型：服务端每接受一个客户端连接，就分配一个独立线程去处理。这个线程在 `read()` 或 `write()` 时会被阻塞——数据没到，线程就干等着，什么都做不了。

这个模型在连接数少的时候没问题。但一旦并发量上来，问题就暴露了：

- **线程数爆炸：** 1000 个连接就需要 1000 个线程，每个线程占用约 1MB 栈内存
- **CPU 利用率极低：** 线程大部分时间在等待 IO，真正做计算的时间不到 10%
- **上下文切换开销：** 操作系统在线程间频繁切换，光切换本身就消耗大量资源

BIO 的本质是**用线程换并发**，但线程是昂贵的资源。当连接数超过几千，服务器就扛不住了。

这就是著名的 **C10K 问题**——一台服务器如何同时处理一万个连接？BIO 模型回答不了这个问题。

## 你的 AI Coding，也是 BIO

现在把"线程"换成"你的注意力"，把"连接"换成"AI 任务"，你会发现大多数人的 AI Coding 和 BIO 一模一样：

**任务级阻塞：** 告诉 AI "帮我写个登录功能"，然后盯着它一步步生成，等它完全完成才开始下一个任务。你的"注意力线程"被这个任务独占。

**流程级阻塞：** AI 写完了，你开始逐行审查。看变量名对不对，看逻辑有没有 bug，看风格统不统一。审查不完，不敢开始下一个任务。就像 BIO 里的线程在 `read()` 上阻塞等数据，你在审查上阻塞等确认。

**注意力级阻塞：** AI 在工作，但你不敢走开。怕它跑偏，怕它理解错需求，怕它生成一堆垃圾代码。你的注意力被完全占住，想做别的事也做不了。

这三种阻塞叠加在一起，你的时间利用率可能不到 30%。

这里有一个反直觉的事实：**AI Coding 的效率瓶颈从来不是 AI 的生成速度，而是你工作流中的阻塞程度。** 就像 BIO 服务器的瓶颈不是网速慢，而是线程被一个连接卡死了——你的瓶颈也不是 AI 不够快，而是你的工作方式让你一直在等。

GitHub 2024 年的调查显示，97% 的开发者已经在使用 AI 编码工具，使用 Copilot 的开发者生产力提升了最高 55%。但大多数人的用法还停留在"一次一个任务，全程盯着"的阶段。工具已经进化了，工作流没有跟上。

## 诊断：你是不是还在用 BIO？

对照一下，如果以下三条中了两条，你就在 BIO 模式：

- 每次只给 AI 一个任务，等它完成才开始下一个
- AI 写完代码后，你逐行审查，比自己写还累
- AI 工作时你全程盯着屏幕，不敢离开

我曾经三条全中。

## NIO 的启示：Selector 和多路复用

Java NIO（Non-blocking IO）解决 C10K 问题的核心武器是 **Selector**（多路复用器）。

Selector 的工作方式是：一个线程注册多个 Channel（连接），然后 Selector 轮询这些 Channel——哪个 Channel 的数据 ready 了，就处理哪个。没 ready 的就跳过，不阻塞。

这背后的架构模式叫 **Reactor 模式**：一个事件循环线程，通过 `epoll`（Linux）或 `kqueue`（macOS）等系统调用，高效监控成千上万个连接的状态变化。Nginx、Node.js、Netty 都是这个模式的实现。

NIO 的线程模型从 BIO 的 **1:1**（一个连接一个线程）变成了 **N:1**（多个连接一个线程）。关键转变是：**线程不再傻等，而是只在数据 ready 时才工作。**

把这套思想映射到 AI Coding：

| IO 概念 | AI Coding 对应 |
|---------|---------------|
| 线程 | 你的注意力 |
| 连接 | AI 任务 |
| Selector | 你的任务管理策略 |
| Channel ready | AI 返回结果 |
| 阻塞等待 | 盯着屏幕等 AI |

**NIO 模式的 AI Coding 意味着：** 你不再被一个任务阻塞，而是同时管理多个 AI 任务，哪个 ready 了就处理哪个。

## 升级到 NIO：三条路径

### 1. 多路复用：同时发起多个任务

BIO 模式下，你一次只给 AI 一个任务。NIO 模式下，你可以同时开多个 Claude Code 会话，发起多个任务，轮流处理结果。

具体做法：

- 开 2-3 个 Claude Code 会话
- 第一个会话让 AI 写用户模块
- 第二个会话让 AI 写订单模块
- 第三个会话让 AI 写测试用例
- 哪个先完成就先处理哪个，不用等其他的

这就像 NIO 的 Selector 同时监听多个 Channel——你不需要为每个任务分配一个"等待线程"，而是用一个管理策略同时处理多个任务。

**关键心态转变：** 你不再是"等 AI 做完"，而是"管理多个 AI 工作者"。就像从 BIO 的"一个连接一个线程"升级到 NIO 的"一个线程管理多个连接"。

### 2. 选择性审查：只看关键的，自动拦截低级的

BIO 模式下，你逐行审查 AI 生成的每一行代码。这就像 BIO 的线程必须亲自处理每一个字节——没有筛选机制，没有优先级。

NIO 模式下，你把审查分成两层：

**人工审查（只看关键部分）：**
- 接口定义和数据结构
- 核心业务逻辑
- 边界条件和错误处理

**自动审查（让工具拦截低级问题）：**
- 用 Prettier / ESLint 自动格式化和 lint
- 用 Claude Code 的 hooks 机制，在每次修改后自动运行检查
- 用自动化测试覆盖常规场景

具体做法：

```json
// .claude/settings.json
{
  "hooks": {
    "postToolUse": [
      {
        "matcher": "Edit",
        "command": "npm run lint -- --fix $CLAUDE_FILE_PATH"
      }
    ]
  }
}
```

这样，AI 每次修改代码，lint 自动运行，低级问题自动修复。你只需要审查逻辑层面的问题。

这就像 NIO 的 Selector 只在 Channel ready 时才通知线程处理——你的注意力只在"真正需要人工判断"时才介入，其他时候让自动化工具处理。

### 3. 异步等待：让 AI 后台工作，你去做别的事

BIO 模式下，AI 工作时你必须盯着。这就像 BIO 的线程在 `read()` 上阻塞——你的注意力被锁死在一个任务上。

NIO 模式下，你可以让 AI 在后台运行，自己去做别的事。

Claude Code 支持 `run_in_background`，你可以：

- 让 AI 在后台跑测试
- 让 AI 在后台重构一个模块
- 你去做 code review、开会、甚至摸鱼
- AI 完成后会通知你

具体做法：

```
> 帮我在后台跑一下这个模块的单元测试，我先去 review 别的 PR
```

**关键心态转变：** 你的注意力不再被 AI 占住，而是被释放出来做更有价值的事。就像 NIO 的线程在等待 IO 时可以去处理其他 ready 的连接，你在等 AI 结果时可以做其他有价值的工作。

## BIO vs NIO：一图看懂

| 维度 | BIO 模式 | NIO 模式 |
|------|----------|----------|
| 任务处理 | 一次一个，串行等待 | 多路复用，轮流处理 |
| 审查方式 | 全程人工逐行审查 | 关键人工 + 自动审查 |
| 注意力占用 | 全程盯屏，不敢离开 | 异步等待，注意力释放 |
| 线程模型类比 | 1:1（一个连接一个线程） | N:1（多个连接一个线程） |
| 时间利用率 | ~30% | ~70% |
| 心态 | "等 AI 做完" | "管理 AI 工作者" |

## 从今天开始升级

不需要一步到位。现在就打开你的终端，做这一件事：

```bash
# 开两个 Claude Code 会话
# 会话 1：让它写一个功能
# 会话 2：让它写另一个功能
# 然后交替处理两个会话的结果
```

就这一步。体验一下"多路复用"的感觉，你会发现等待的时间突然变少了。

然后这一周，逐步加上另外两步：

1. **这周：** 给项目配好 lint 和自动测试，让 AI 改完代码后自动检查
2. **下周：** 试一次让 AI 在后台跑测试，你去做别的事

每一步都在减少阻塞，每一步都在释放你的时间。

**AI Coding 的效率瓶颈从来不是 AI 的速度，而是你工作流的阻塞程度。** 从 BIO 到 NIO，改变的不是工具，而是工作流。工具已经进化了，你的工作流也该跟上了。
